production hardening case study

⚡ PRODUCTION HARDENING · NEBULARK INC.
Scaling a Live Game for High CCU
// full script audit + rewrite · concurrency safety · datastore architecture · Mar 2026
The game was well-architected from day one with solid systems thinking and clean module separation. This session was purely about hardening it for production scale. The issues found were the kind that only surface under real concurrency — they don't show up in solo Studio testing. Eight independent PlayerAdded listeners all racing to read profile data that might not exist yet, player objects used as table keys instead of UserIds causing memory leaks under churn, and _G as a data bus becoming unpredictable when multiple scripts fire simultaneously. A coroutine queue deadlock in PlayerBases was a subtle Lua-specific issue where coroutine.running() returns nil when called outside a coroutine — something that only surfaces when the calling context changes. The result is a codebase that now handles real player load safely.
⚠ before
  • 8 independent PlayerAdded listeners racing on join
  • Player objects as table keys — memory leaks under load
  • _G used as cross-module data bus — race conditions on join
  • Coroutine queue deadlock blocking base assignment
  • SetAsync fired on every coin flip — datastore hammering
  • Full workspace traversal on every brainrot spawn
  • No concurrency locks on place / pickup / upgrades
  • Username + thumbnail HTTP calls on every board refresh
✓ after
  • Single authoritative join sequence, guaranteed order
  • UserId keys everywhere — no leaks, clean on leave
  • DataStore.GetProfile(userId) module API — no globals
  • task.wait() mutex — works from any calling context
  • Flips batched to ProfileService auto-save on leave
  • BrainrotCache built once at startup — O(1) at runtime
  • Per-stand concurrency locks on all write operations
  • Username + thumbnail cached per UserId forever
